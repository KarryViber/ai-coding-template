## Explore 模式规则

### 调研优先，代码其次

这个模式的主要输出是**方案/结论**，不是代码。

1. **定义问题** — 要调研什么？判断标准是什么？
2. **多方调研** — 搜索方案，找开源项目，看技术对比
3. **方案对比** — 每个方案的优劣、适用场景
4. **给出推荐** — 明确推荐哪个，为什么
5. **POC（可选）** — 如果需要验证可行性，写最小 POC

### 输出格式

调研结果按以下格式整理：

```
## 调研：[主题]

### 背景
- 要解决什么问题

### 方案对比
| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| A    | ...  | ...  | ...      |
| B    | ...  | ...  | ...      |

### 推荐
- 推荐方案: ...
- 推荐理由: ...

### 下一步
- 如果采纳，接下来做什么（可切换到 Build 模式）
```

### Agent Team 协作

本模式预配置了 agent team：

- `researcher-a` — 负责一个调研方向
- `researcher-b` — 负责另一个调研方向

适合需要对比多个方案的场景。如果调研问题比较聚焦，用单 agent 就够了。

## AI 协作规则

### 工作流程

- 不急着写代码 — 先把方案搞清楚
- 给出结论和推荐，不要只罗列信息
- 对比时要有判断，不要两边都说好
- 如果信息不足，主动说"需要进一步调研"

### 代码风格（POC 时）

- POC 代码只求能跑，不求完美
- 注释可以多一点 — POC 代码是用来说明思路的
- 不需要完整的错误处理

### Git

- 不要自动 commit — 先问我要不要提交
- 使用 conventional commits + emoji

### 验证检查点

调研完成后：
1. 确认所有调研问题都回答了
2. 方案对比是否公平、全面
3. 推荐理由是否站得住脚
