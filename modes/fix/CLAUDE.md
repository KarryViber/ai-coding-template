## Fix 模式规则

### 修复前三问

每次修 bug 前，先回答：

1. **根因是什么？** — 不是现象，是底层原因
2. **修复如何解决根因？** — 不是打补丁，是真正修掉
3. **怎么验证修好了？** — 完整的验证步骤

### 调试流程

1. **复现问题** — 先确认自己能稳定复现
2. **定位根因** — 缩小范围，找到出问题的具体代码
3. **说明方案** — 告诉我根因和修复思路，确认后再改
4. **实施修复** — 改代码
5. **完整验证** — 跑完整用户流程，不只是改动点

### 反补丁原则

- 不要打补丁 — 修根因
- bug "重现"时，先查原始诊断是否正确
- 如果修一个 bug 引入了新 bug，回退重新思考
- 同一个问题修两次还没好，停下来重新分析

### 单 Agent 模式

Fix 模式不使用 agent team。调试需要深度专注和连续的上下文，分散到多个 agent 反而低效。

### 推荐 Skills

# 完成前验证
claude skill install superpowers:verification-before-completion

## AI 协作规则

### 工作流程

- 修复前必须先说思路，确认后再改
- 改完代码后：确认能跑通了再说搞定
- 发现潜在问题主动提醒，即使我没问

### 代码风格

- 注释尽量少 — 只在逻辑不自明的地方加
- 不要加 JSDoc/docstring，除非我要求
- 不要顺手重构 — 只修 bug
- 不要碰我没让你改的代码

### Git

- 个人项目：直接提交到 main
- 不要自动 commit — 先问我要不要提交
- 使用 conventional commits + emoji（🐛 fix 为主）

### 验证检查点

每次修复后：
1. 确认修复生效了
2. 用原始复现步骤验证
3. 测试相关的正常流程没被破坏
4. 如果涉及边界情况，测试边界
